Fifty Fifty

A game to simulate life and various events. While the 'game' is simply a few buttons the main focus is building a customizable editor to handle easy creation and handling of Scriptable Objects.

Overview and Goals:

Create simple editor modules capable of loading and unloading each other as well as handling saving and loading of 'Scriptable Objects'. Editor should be intuitive and allow mass creation easily.

Finally create a large pool of events that the player could potentially come into contact with and filter that pool by what the character has been doing recently, what skills they have etc.

The 'Game' portion itself would play very similarly to BitLife or other 'Life Simulation' games that use a text adventure style.

Editor Modules:

Above is the blank editor modules, these are used to create the pool of events. 'Events' and 'Questions' are synonymous.

Answer Builder

Answers are possible ways the player can respond to a given question. They can include their own requirements, however many are blanket responses to situations.

Requirements allow personalisation, such as high strength characters being able to fight their way out of situations; simply add a requirement that a character must have x strength.

Answers can also require contacts, items, or previous events. This allows chaining for storylines to emerge organically through gameplay, but given direction so they can recieve proper polish.

Question Builder

Questions are the situations the player can get into. These can be as wide ranging as one wishes. Requirements are how the player is limited from seeing questions, this is so that a baby cannot be given an event to buy a house, for example.

Questions must feature at least 2 responses which have equal or lower requirements than the actual question; this is to prevent the player recieving answers they shouldn't be able to perform. All of this is handled in the back end so the designer doesn't have to worry about it.

Questions however can include as many answers as they want after these two with any requirements; these will then be compared with the player if they receive the event and be added into the pool if the player passes.

Currently questions will only display 2 of the answers out of the questions however this could be easily expanded to n amount or to all the player passes.

Requirements Builder

This is the core of how all the pooling works and how a player is navigated through the world by their choices. Requirements include things such as Contacts (people they need to have met), Items (items they must have acquirred) as well as other events (Questions), including answers to those questions (so the player has to recieve a question and respond in a certain way).

Next Steps:

Actually use the tools I made... A small UI was created for the game but mostly I have been working with a basic object that I change to debug and test. I created a lot for this project but eventually felt like I created so much it deserved to be used for something else. It also happened that ECS came out around this time, I learnt of BitLife which was a very similar idea, I was disheartened and lost interest in the project.

In loading up the project again though I discovered everything still works and therefore if I wished for this project to continue, I'd certainly work on preview sprites being default in the editor, as well as working more on the layout, while the dropdowns are nice something to seperate categories more distinctly might be better.
